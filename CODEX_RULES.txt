Codex-руководство PhuketRide — обновлённый набор правил для агента управления арендой автомобилей:

I. Взаимодействие
1. Отвечать кратко, по делу и строго на русском языке.
2. Контролировать контекст разницы между зонами приложения и сохранять целостность архитектуры, чтобы проект оставался связанным.
3. Ошибки валидации и служебные действия показывать только через toast-уведомления без повторов в других формах.
4. Минимизировать лишние шаги: выполнять только действительно нужные действия и параметры, не усложнять поток.
5. Ограничить количество строк в файле до 500.

II. Работа с данными и БД
1. Использовать исключительно Cloudflare D1 на production — владение БД через `wrangler d1 execute <db-name> --remote`.
2. Запускать `wrangler` всегда с флагом `--remote`, локальные файлы SQLite из `.wrangler/state/v3/d1/*.sqlite` запрещены.
3. После любой работы удалять временные и служебные файлы, включая `.wrangler/state`, чтобы не оставлять локальный инс-тейд.
4. При необходимости самостоятельно формировать и выполнять SQL-запросы.

III. Файлы и структура проекта
1. Не создавать файлы в формате `.md` без прямого разрешения.
2. Все иконки брать только с https://heroicons.com.
3. Компоненты админки держать только в `app/components/dashboard`, а клиентские — только в `app/components/public`; запретить пересечение и переиспользование между этими ветками.
4. Учитывать архитектуру и контекст в разных частях проекта, чтобы изменения не разъединяли их.

IV. Ограничения на команды и запуск
1. Команды `npm run dev` и `npm run build` нельзя выполнять без явного разрешения.
2. Действия выполняются по запросу: запускаем только то, что запросил пользователь, и только если это не нарушает других правил.

V. Качество и безопасность
1. Безопасность сессии обязательна: в production cookie сессии должны быть `secure=true`, `httpOnly=true`, `sameSite=lax` (или строже); изменения `sessionCookie` допускаются только с учетом HTTPS.
2. Единая серверная валидация: все `action` и `loader` обязаны валидировать входные данные через `zod` и схемы из `app/schemas`; разрозненная ручная валидация не допускается.
3. Rate limit обязателен для `login`, `register` и публичных API: соответствующие маршруты должны вызывать централизованную проверку лимитов.
4. RBAC по умолчанию: каждый dashboard-маршрут обязан явно вызывать `requireAuth` или `requireRole`; отсутствие проверки доступа считается нарушением.
5. Аудит критичных операций обязателен: `create`, `update`, `delete`, `export` логируются через `quickAudit` с `userId`, `entityType`, `action` и метаданными запроса.
6. SQL только параметризованный: использовать `prepare().bind()`; SQL-конкатенации и интерполяции строк в запросах запрещены, включая служебные скрипты.
7. UX ошибок единообразный: ошибки и служебные сообщения показываются только через toast; дублирование inline-блоками в формах запрещено.
8. Перед деплоем обязателен `npm run typecheck`; деплой без успешного typecheck запрещен.

VI. Документация проекта
1. Перед изменением логики сначала сверяться с документацией в `docs/README.md`.
2. Основные ссылки на документацию проекта:

VII. Завершение задач после изменений
1. После любых изменений в коде обязательно проверить работоспособность затронутых страниц и сценариев.
2. Завершать задачу только если страницы открываются без ошибок и базовая проверка пройдена.
